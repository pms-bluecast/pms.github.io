1. Introduction

Purpose: This document defines the standard enterprise software delivery process for:

  • Web applications, Mobile applications, APIs & Microservices, Enterprise platforms
  • AI/ML applications, SaaS products, Internal systems

Objectives: Predictable delivery, High-quality software, Reduced rework, Better scalability, Security compliance, Clear accountability, Faster releases, Better maintainability, Improved collaboration.

🔴 TIER 1 REQUIREMENT: All phases are MANDATORY. No phase can be skipped without a signed waiver from CTO and Client.

2. Core Principles

Clarity Before Coding Design Before Implementation Single Source of Truth Shift Left Quality Security by Design Scalability by Default Reusability First Automation First Observability Mandatory Controlled Change Management

Shift Left Quality Approach: Testing and quality assurance activities are moved to the earliest possible stages — catching issues during requirements and design phases rather than reacting to bugs late in production.

5. Requirement Engineering

Objective: Convert business expectations into structured, testable, and measurable requirements.

RASCI Model

📚 Industry Best Practices
  • Use SMART criteria for requirements (Specific, Measurable, Achievable, Relevant, Time-bound)
  • Apply MoSCoW prioritization (Must-have, Should-have, Could-have, Won't-have)

  • Perform impact analysis for every requirement change
  • Include Non-Functional Requirements (NFRs) in SRS
  • Validate requirements with prototypes or wireframes
  • Apply INVEST principle (Independent, Negotiable, Valuable, Estimable, Small, Testable)
✅ Phase Checklist

  • All functional, non-functional, and security requirements documented

  • Data flow and dependencies identified
  • User & Access Governance defined (RBAC)

7.7 Reusable Component Design

7.8 Security Design

Define: Authorization checks, Role validations, Input sanitization, Encryption, Token handling, Session management

7.9 Performance Design

Deliverables: Query Optimization Plan, Caching Strategy, API Optimization Design, Async Processing Design, Frontend Performance Plan, Scalability Strategy, Queue Design, Monitoring Metrics Design

7.11 State Management Design

Define: Data flow, State persistence, Cache handling, Synchronization

📚 Industry Best Practices
  • Apply SOLID principles and design patterns consistently
  • Use database normalization (3NF) with denormalization for performance where needed
  • Implement ETVX (Entity-Transaction-Validation-eXception) for business logic
  • Design for idempotency in all write operations
  • Apply pessimistic/optimistic locking based on concurrency requirements
  • Use structured logging with correlation IDs for distributed tracing
  • Implement retry with exponential backoff for transient failures
  • Design APIs with pagination, filtering, sorting, and field selection
✅ Phase Checklist
  • ER Diagram and DB schema design completed
  • Database optimization (indexes, partitioning, sharding) planned
  • ETVX document for business logic completed
  • API Specification (OpenAPI) completed and approved by all teams
  • UI/UX designs (wireframes, prototypes) approved
  • Security design (encryption, token handling, session management) documented
  • Performance design (caching, async, queue) documented
  • Logging & audit specifications defined
  • State management design (frontend/mobile) completed
  • Unit test scenarios and boundary conditions documented

  • Apply Test-Driven Development (TDD) or Behavior-Driven Development (BDD)
  • Use pre-commit hooks for linting, formatting, and secret detection
  • Follow Conventional Commits specification for commit messages
  • Implement peer review with mandatory approval before merge
  • Maintain code coverage threshold (≥80%) for all modules
  • Apply clean architecture and dependency injection
  • Never hardcode secrets; use secret management always
  • Implement health check endpoints for all services
✅ Phase Checklist
  • Code planned and designed before coding (including unit test cases)
  • Functional requirements covered (ETVX, sequence diagrams, flowcharts)
  • Unit testing completed and documented
  • Integration tests completed before deployment
  • Smoke tests (UAT cases) passed before QA release
  • All coding standards and naming conventions applied
  • Date/time functions standardized (UTC) across project
  • Dead code and console logs removed before QA release
  • Peer review completed and signed off
  • No hardcoded secrets (all in Vault/Secrets Manager)

10. Performance Testing

Activities: API latency validation, Load testing, Infrastructure health check

ResponsibleAccountableSupportiveConsultedInformed
QADevOpsDev teamArchitect, CTOClient
📚 Industry Best Practices
  • Test with production-like data volume and traffic patterns
  • Establish baseline performance metrics before load testing
  • Test for 2x-3x expected peak traffic
  • Monitor resource utilization during tests (CPU, Memory, I/O, Network)
  • Identify and optimize slow queries (N+1 problem)
  • Test database connection pool limits
  • Verify caching effectiveness (hit ratio)
  • Test queue processing throughput and lag
✅ Phase Checklist
  • API latency validated against SLA (p95, p99)
  • Load testing completed at 2x expected traffic
  • Stress testing completed (breakpoint identified)
  • Soak testing completed for memory leak detection
  • Scalability testing (horizontal scaling) validated
  • Database query performance optimized (indexes added)
  • Caching strategy effectiveness verified
  • Queue processing latency within acceptable range
  • Infrastructure auto-scaling configured and tested
  • Performance test report delivered and approved

Deliverables: VAPT report

ResponsibleAccountableSupportiveConsultedInformed
QADevOpsDev teamArchitect, CTOClient
📚 Industry Best Practices
  • Apply OWASP Top 10 as minimum security baseline
  • Implement SAST in CI/CD pipeline (fail on critical findings)
  • Perform DAST on staging environment before production
  • Scan dependencies for known vulnerabilities (Snyk, Trivy)
  • Conduct penetration testing for critical modules
  • Validate all authentication and authorization controls
  • Test for rate limiting and brute force protection
  • Verify audit logs for all security events
✅ Phase Checklist
  • SAST (Static Analysis) completed - no critical findings
  • DAST (Dynamic Analysis) completed
  • Dependency scan completed - no critical CVEs
  • Vulnerability Assessment & Penetration Testing (VAPT) completed
  • RBAC validation completed for all roles
  • Audit logging enabled and verified
  • Secrets scan completed - no hardcoded secrets
  • Input sanitization and encryption validated
  • Rate limiting implemented on all API endpoints
  • Security test report delivered and approved

12. Deployment Phase

Main Goals: Safe releases, Minimal downtime, Predictable deployments, Faster rollback, Stable environments, Traceability, Zero configuration drift

🚫 NEVER Deploy Without: QA approval, Backup, Rollback plan, Monitoring, Security validation, Smoke testing
🚫 NEVER Allow: Direct production code edits, Unapproved releases, Hardcoded secrets, Untested DB migrations

Major Deployment Activities:

  • Build Preparation, Code Freeze & Release Branching, CI/CD Pipeline Execution
  • Environment Preparation, Configuration Management (HashiCorp Vault)
  • Infrastructure Deployment (IaC: Terraform, Helm)
  • Database Migration Execution (backward compatible, rollback support)
  • Application Deployment (Rolling/Blue-Green/Canary)
  • Smoke Testing, Integration Validation, Monitoring Setup
  • Backup & Rollback Preparation, UAT/Staging Validation
  • Production Release Approval, Production Deployment
ResponsibleAccountableSupportiveConsultedInformed
DevOpsTech leadsDev teamQA, Architect, CTOClient

📚 Industry Best Practices
  • Use Blue-Green or Canary deployment for zero-downtime releases
  • Implement Infrastructure as Code (Terraform, Helm)
  • Maintain immutable infrastructure (never patch running containers)
  • Use feature flags for progressive rollout
  • Automate rollback with one-click capability
  • Validate all health checks before traffic shift
  • Monitor deployment metrics in real-time
  • Maintain deployment runbook with step-by-step instructions
✅ Phase Checklist
  • QA Sign-off obtained for staging
  • Security Sign-off obtained
  • Performance Sign-off obtained
  • Product Owner Sign-off for production
  • Database backup completed before migration
  • Rollback procedure tested and documented
  • Smoke test scenarios prepared and executed
  • Monitoring dashboards and alerts configured
  • Release notes prepared and communicated
  • Incident escalation plan active and on-call notified

26.1 Docker Best Practices

🐳 Dockerfile Best Practices
# Multi-stage build for optimal image size FROM node:20-alpine AS builder WORKDIR /app COPY package*.json ./ RUN npm ci --only=production && npm cache clean --force COPY . . RUN npm run build FROM node:20-alpine AS runner RUN addgroup --system --gid 1001 nodejs RUN adduser --system --uid 1001 appuser WORKDIR /app COPY --from=builder --chown=appuser:nodejs /app/dist ./dist COPY --from=builder --chown=appuser:nodejs /app/node_modules ./node_modules USER appuser EXPOSE 3000 HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \ CMD node health-check.js || exit 1 CMD ["node", "dist/main.js"]

Docker Best Practices Checklist:

  • ✓ Use official base images (alpine variants for smaller size)
  • ✓ Multi-stage builds to reduce final image size
  • ✓ Run as non-root user (create appuser)
  • ✓ Set HEALTHCHECK instruction for container orchestration
  • ✓ Use .dockerignore to exclude unnecessary files
  • ✓ Layer caching optimization (copy package.json before source)
  • ✓ Label images with version, build date, and git commit
  • ✓ Scan images for vulnerabilities (Trivy, Snyk)
  • ✓ Never store secrets in images (use secrets management)

27. 🤖 AI Services Guidelines & Checklist

Purpose: Ensure AI/ML services are developed, deployed, and governed with the same rigor as critical software, with additional considerations for model quality, bias, and prompt security.

27.1 Model Training & Development

Guidelines for building, training, and validating AI/ML models internally
📚 Industry Best Practices
  • Version Control: Use DVC, MLflow, or Git LFS for model and dataset versioning
  • Dataset Provenance: Document data source, collection method, preprocessing, and version
  • Data Quality: Validate data completeness, accuracy, and representativeness
  • Reproducibility: Record all hyperparameters, training environment, and random seeds
  • Bias Testing: Evaluate for demographic parity, equal opportunity, and disparate impact
  • Explainability: Implement SHAP, LIME, or attention visualization for model decisions
  • Performance Tracking: Monitor accuracy, precision, recall, F1, latency, throughput
  • Model Card: Create comprehensive model documentation (Mojo and Huyen's framework)
  • Cross-validation: Use k-fold cross-validation to prevent overfitting
  • Test/Train Split: Maintain 70/20/10 split (Train/Validation/Test)
✅ Phase Checklist
  • Model versioning implemented (DVC/MLflow) MANDATORY
  • Training dataset documented with source, version, preprocessing MANDATORY
  • Data quality assessment completed (completeness, accuracy, representativeness)
  • Hyperparameters documented with justification
  • Training environment (hardware, OS, dependencies) documented
  • Bias validation completed (demographic parity, equal opportunity) MANDATORY
  • Explainability/interpretability implemented GDPR
  • Model Card created and reviewed MANDATORY
  • Cross-validation completed (k-fold or stratified)
  • Test set performance validated against acceptance criteria
  • Model performance drift monitoring defined
  • Data retention and privacy compliance verified (GDPR, CCPA)

📋 Model Card Requirements (Mandatory)

  • • Model Details: Name, version, architecture, training date, authors
  • • Intended Use: Primary use case, intended users, out-of-scope uses
  • • Training Data: Source, size, preprocessing, limitations
  • • Evaluation Data: Test set description, evaluation methodology
  • • Performance Metrics: Accuracy, precision, recall, F1, latency, throughput
  • • Limitations: Known limitations, edge cases, failure modes
  • • Bias Analysis: Group fairness metrics, known biases, mitigation strategies
  • • Ethical Considerations: Impact assessment, deployment constraints
⚠️ CRITICAL: Models with bias scores exceeding acceptable thresholds (Δ < 0.1) MUST NOT be deployed to production. Bias mitigation techniques must be applied before deployment.

27.2 Model Subscription & Use

Guidelines for procuring, integrating, and using third-party AI/ML services
📚 Industry Best Practices
  • Vendor Assessment: Evaluate model provider's compliance, security, and privacy policies
  • API Key Management: Store API keys in Vault/AWS Secrets Manager - NEVER in code
  • Rate Limiting: Implement RPM (Requests Per Minute) and TPM (Tokens Per Minute) limits
  • Prompt Versioning: Store all prompts in version control with review history
  • Prompt Injection Protection: Sanitize inputs, escape delimiters, separate instructions
  • Content Filtering: Implement profanity filters, PII detection, harmful content blocking
  • Response Logging: Log all AI responses with user ID, timestamp, prompt, model version
  • Cost Monitoring: Track token usage and cost per user/feature
  • Fallback Strategy: Define graceful degradation when model unavailable or confidence low
  • Human-in-the-Loop: Critical decisions require human approval and override capability
✅ Phase Checklist
  • Vendor security assessment completed MANDATORY
  • API keys stored in Vault/Secrets Manager (not in code) MANDATORY
  • Rate limiting configured (RPM, TPM) MANDATORY
  • Prompt versioning implemented MANDATORY
  • Prompt injection protection implemented MANDATORY
  • Content filtering and moderation active
  • All AI responses logged with request context MANDATORY
  • Cost tracking and budget alerts configured
  • Fallback mechanism defined for model failure
  • Human approval workflow for critical decisions MANDATORY
  • Data privacy and processing agreements in place
  • Model output validation pipeline defined

🔐 Security & Privacy Requirements

  • • Data Privacy: Do NOT send PII, PHI, or sensitive data to external AI APIs without explicit approval
  • • Data Retention: Logs containing prompts must be retained for minimum 30 days, maximum 90 days
  • • Anonymization: User identifiers in prompts must be anonymized before logging
  • • Model Evaluation: Test model outputs on a sample dataset before full deployment
  • • Guardrails: Define output validation rules (length limits, banned patterns, format validation)
⚠️ CRITICAL: Any use of external AI services requires prior approval from the Security & Privacy Committee. Vendors must be reviewed for compliance with GDPR, EU AI Act, and organizational data protection policies.
